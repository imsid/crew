# Mash Crew: Internal-First Agentic Workspace for PM + GTM

## Summary
Build `mash-crew` into a web-first, internal product for PM + GTM teams where agents do three things well: ground every answer in trusted business metrics, turn analysis into reusable team artifacts, and execute bounded workflows with human approval. The current repo already has the right kernel: role agents, subagent delegation, session memory, telemetry, SSE streaming, and a metrics compiler. The gap is product shape, not raw agent capability.

The v1 product should be a collaborative workspace, not a smarter CLI. Users interact through a ChatGPT-like thread UI with embedded charts/tables, artifact panels, approvals, and reusable workflow templates. The metrics layer remains the source of truth for semantic metrics; the crew app sits on top of it and turns those metrics into decisions, briefs, readouts, and follow-up actions.

## Product Direction
- Make PM + GTM the first wedge. Focus on workflows where both teams already overlap: launch planning, funnel diagnosis, experiment readouts, weekly business reviews, and voice-of-customer synthesis.
- Optimize for bounded execution. Agents can gather evidence, run analyses, draft deliverables, update workspace artifacts, and request approvals before any external or sensitive action.
- Use a web app as the primary surface. Keep the existing Mash host as the execution engine and add a thin crew application layer for threads, artifacts, workflows, and collaboration.
- Treat every useful output as a first-class artifact. Good answers should become reusable team assets instead of disappearing in a chat log.
- Make visualizations native. Every metric-backed answer should be able to render as a chart card, KPI card, table, or saved dashboard section.
- Design around shared workflows, not isolated personas. Marketing analyses should be reusable by PM; PM launch artifacts should be reusable by GTM; support insights should feed product discovery.

## Core Experience
- `Workspace`: team-scoped home for threads, saved artifacts, dashboards, metric collections, and workflow runs.
- `Thread`: chat session mapped onto Mash `session_id`, with streaming execution, visible agent handoffs, and embedded structured outputs.
- `Artifact`: durable output generated by agents or humans. Initial kinds: `analysis`, `launch_brief`, `metric_readout`, `dashboard_view`, `experiment_review`, `customer_insight_pack`, `action_plan`.
- `Workflow`: multi-step run that combines prompts, metrics, evidence gathering, artifact creation, and approval gates. Initial templates:
  - Weekly business review
  - Launch readiness + launch readout
  - Funnel drop diagnosis
  - Experiment result review
  - VOC synthesis from support/research notes plus metrics
- `Approval`: explicit human checkpoint for actions that change shared state, publish outputs, or trigger downstream systems.
- `Evidence model`: every substantive claim should attach `metric_refs`, `source_refs`, and `confidence` so users can inspect provenance.

## Architecture and Mashpy Mapping
- Keep `crew.app:build_host` as the agent runtime entrypoint.
- Add a `crew` application API layer in front of the Mash host. It should own workspace metadata, artifact persistence, workflow templates/runs, approval state, and thread composition.
- Use existing Mash host APIs directly for:
  - `/agent/{id}/request` and SSE event stream for async chat execution
  - session history for thread replay
  - preferences for per-user and per-workspace defaults
  - app-data for run-scoped structured state until a shared store is added
  - telemetry and memory search for run inspection and debugging
  - subagent routing for PM -> data -> engineer collaboration
- Introduce a shared app store for cross-agent artifacts and workflow state. Current per-agent SQLite state is not enough for true collaboration.
- Standardize agent outputs into structured blocks:
  - `text`
  - `metric_card`
  - `chart_spec`
  - `table`
  - `artifact_ref`
  - `approval_request`
  - `next_action`
- Keep the metrics layer compile-first contract intact. Data agents must answer through metric definitions and compiled SQL plans, not ad hoc warehouse semantics.
- Add an execution policy layer:
  - read-only analysis actions run automatically
  - artifact writes are allowed in the workspace
  - external side effects require approval
- Fix the current host inconsistency and fully register `engineer` as a subagent so product, data, and engineering workflows can all participate in one crew.

## Metrics Layer as Product Backbone
- Expand the metrics layer from “compiler” to “trust product”:
  - metric ownership
  - certification status
  - freshness metadata
  - dimension definitions
  - lineage to source configs and compiled SQL
  - business definitions and caveats
- Add a curated metric catalog UI that business users can browse before asking questions.
- Require every metric-backed answer to expose the exact metric IDs used, the chosen dimensions/filters/date range, and a drill-down path to the compiled SQL plan.
- Add saved semantic query recipes for common PM + GTM questions so repeated workflows are faster and more consistent.
- Add metric health checks for broken configs, stale data, and incompatible dimension usage; surface those directly in the app so trust issues are visible, not hidden.

## Collaboration Model
- Make artifacts shareable, commentable, and remixable across teams.
- Allow users to fork an artifact into a new thread or workflow, preserving context and evidence.
- Add “handoff” as a first-class action: PM can hand an analysis to GTM, GTM can request a PM framing, data can attach a grounded readout, engineer can assess implementation impact.
- Support shared dashboards and recurring workspace views for weekly reviews and launch rooms.
- Preserve agent provenance in collaborative threads so users know which agent produced which insight and which evidence it used.

## Mashpy Improvements to Prioritize
- First-class shared memory/artifact store across agents rather than only per-agent state.
- Structured response block schema so clients do not parse free-form text to render charts, tables, approvals, and artifacts.
- Long-running workflow runtime with resumable steps, status, retries, and approval waits.
- First-class approval task primitive.
- Better multi-agent run tracing so UI can show delegation, tool calls, and result stitching cleanly.
- Optional policy hooks for action gating, PII boundaries, and tool authorization by workspace/user role.

## Public Interfaces to Add
- App API resources:
  - `Workspace`
  - `Thread`
  - `Artifact`
  - `WorkflowTemplate`
  - `WorkflowRun`
  - `ApprovalRequest`
  - `MetricDefinitionSummary`
- Thread response contract:
  - `message_parts[]`
  - `agent_trace[]`
  - `artifact_ids[]`
  - `metric_refs[]`
  - `approval_requests[]`
  - `suggested_followups[]`
- Artifact schema minimum fields:
  - `id`, `kind`, `title`, `summary`, `body`, `created_by`, `source_thread_id`, `metric_refs`, `source_refs`, `status`
- Workflow run schema minimum fields:
  - `id`, `template_id`, `workspace_id`, `inputs`, `step_status`, `artifacts`, `approvals`, `final_summary`

## Delivery Phases
- Phase 1: foundation
  - Fix host registration and agent metadata consistency.
  - Add shared workspace/artifact/workflow models and API.
  - Standardize structured agent outputs.
  - Build the web app shell with threads, streaming responses, artifact side panel, and basic chart rendering.
- Phase 2: PM + GTM wedge
  - Ship the five workflow templates above.
  - Add metric catalog, saved views, and grounded chart/table cards.
  - Add handoff and approval flows.
- Phase 3: collaboration and scale
  - Shared dashboards, recurring review rooms, and richer comments/remix flows.
  - Expand to marketing, sales, support, and research through the same workflow/artifact model rather than separate bespoke apps.

## Test Plan
- Host tests
  - primary/subagent registration includes `pm`, `data`, `engineer`, `masher`
  - subagent metadata is injected and discoverable
- Metrics trust tests
  - every metric-backed response returns metric refs and source refs
  - invalid dimensions or stale metric configs surface deterministic errors
  - compiled SQL lineage matches the artifact metadata shown in UI
- Workflow tests
  - launch readout workflow creates the expected artifact set
  - approval-required steps pause and resume correctly
  - cross-agent handoff preserves context and provenance
- API tests
  - thread streaming, artifact persistence, workflow status, and approval endpoints
  - session-to-thread mapping and history replay
- UI tests
  - streaming thread experience
  - embedded chart/table rendering
  - artifact fork/share flows
  - approval UX
- Acceptance scenarios
  - PM asks for launch readout, gets grounded charts plus a reusable launch artifact
  - GTM forks that artifact into a go-to-market action plan
  - Data agent exposes exact metric definitions used
  - Engineer can be pulled in for implementation feasibility without losing thread context

## Assumptions
- Internal-first deployment for one company or a small set of internal teams.
- PM + GTM is the initial value wedge.
- Primary interface is a web app; CLI remains a secondary power-user/debug surface.
- BigQuery remains the initial warehouse and the metrics layer remains the semantic source of truth.
- Bounded execution is the autonomy model: agents can analyze and draft freely, but external side effects require approval.
- Multi-tenant concerns, complex RBAC, and broad external integrations are deferred until the PM + GTM wedge proves repeatable value.
