# Analyzing User Feedback

Help the user turn qualitative feedback into clear product insights.

## How to Help

When the user asks for help with feedback analysis:

1. **Understand the input set** - Clarify sources, sample size, customer segments, and recency
2. **Separate signal from anecdotes** - Look for repeated patterns, not just memorable quotes
3. **Cluster by problem, job, or moment** - Group feedback into actionable themes
4. **Convert findings into decisions** - Tie themes to product opportunities, risks, or roadmap implications

## Core Principles

### Look for repeated struggles, not isolated opinions

The goal is to identify recurring user pain in context, especially around blocked jobs or unmet expectations.

### Preserve the user's language

Use customer phrasing to avoid translating away the actual problem.

### Segment before synthesizing

Feedback from new users, power users, churned users, and enterprise admins should not be mixed indiscriminately.

### Separate symptoms from root causes

Complaints about a feature often point to a broader workflow issue or expectation mismatch.

### Distinguish severity from frequency

A rare issue can still matter if it is catastrophic for an important segment.

### Tie feedback to a specific product moment

Acquisition, activation, setup, daily use, collaboration, billing, and support all produce different kinds of insights.

## Questions to Help Users

- "What feedback sources are you combining, and from which user segments?"
- "Which themes appear repeatedly across multiple sources?"
- "What exact words are users using to describe the problem?"
- "Is this complaint about usability, trust, missing capability, or expectation mismatch?"
- "Which themes are both frequent and strategically important?"
- "What product decision are you trying to inform with this analysis?"

## Common Mistakes to Flag

- **Overweighting loud anecdotes**
- **Combining unlike user segments**
- **Jumping from quotes to solutions too quickly**
- **Treating sentiment as insight**
- **Ignoring workflow context**

## Suggested Output Structure

1. Top themes
2. Representative user language
3. Affected segments
4. Severity and frequency notes
5. Product implications
